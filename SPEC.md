# 株価監視・急騰通知アプリ 仕様書

最終更新: 2026-03-01

---

## 1. プロジェクト概要

### 目的
日本株（東証）を対象に、場中の急騰銘柄をリアルタイムに近い形で検知し、LINE通知を送るシステム。

### ゴール
- 誤検知を少なく、発報を早くする
- バイブコーディングをしない（仕様を先に決めてから実装する）

### 技術スタック
| 区分 | 技術 |
|---|---|
| 言語 | Python 3.12 |
| UI | Streamlit |
| チャート | Plotly |
| データソース（現在） | yfinance（15分遅延） |
| データソース（将来） | auカブコム証券API（リアルタイム） |
| 通知 | LINE Messaging API |
| スケジューラ | APScheduler |
| テクニカル指標 | pandas-ta |

## 3. 現在の急騰検知ロジック（固定閾値モデル）

### 発報条件

```
(前日終値比 >= 閾値A  OR  当日始値比 >= 閾値B)
              AND
出来高比 >= 閾値C

→ detected = True → 発報
```

### デフォルト閾値（config.py）

| 閾値 | デフォルト値 | 意味 |
|---|---|---|
| PRICE_CHANGE_FROM_PREV_CLOSE | 3.0% | 前日終値比 |
| PRICE_CHANGE_FROM_OPEN | 5.0% | 当日始値比 |
| VOLUME_RATIO | 3.0倍 | 出来高 / 過去20日平均出来高 |

閾値は監視銘柄管理ページのスライダーで銘柄ごとに変更可能。未設定時は config.py のデフォルト値を使用。

### テクニカル指標の現在の扱い

| 指標 | 発報判定 | 画面表示 | 備考 |
|---|---|---|---|
| 前日終値比 | 使用 | 表示 | 判定の主条件 |
| 当日始値比 | 使用 | 表示 | 判定の主条件 |
| 出来高比 | 使用 | 表示 | 判定の必須条件 |
| RSI (14) | 未使用 | 表示 | 参考表示のみ |
| ゴールデンクロス | 未使用 | 表示 | 参考表示のみ |
| BB上抜け | 未使用 | 表示 | 参考表示のみ |

### スキャン仕様

| 項目 | 値 |
|---|---|
| データ取得足種 | 15分足（固定） |
| 取得期間 | period="5d"（前日終値計算に必要） |
| スキャン間隔 | SCAN_INTERVAL_MINUTES（デフォルト: 5分） |
| 取引時間 | TRADING_START_TIME〜TRADING_END_TIME（デフォルト: 09:00〜15:30） |
| 最大遅延 | 約20分（yfinance 15分遅延 + スキャン間隔5分） |

注意: スキャンの足種はチャート分析ページの足種選択と無関係。auカブコム API 対応後に設定可能にする予定。

## 4. UI仕様

### ページ構成

#### チャート分析ページ
- 銘柄コード（4桁）入力
- 足種切り替えボタン（秒足〜週足）
- 表示期間選択（1週間〜1年）
- テクニカル指標ON/OFF（SMA・EMA・BB・RSI・MACD・出来高）
- 現状は参照専用。スキャナーとは無関係。

#### 監視銘柄管理ページ
- 銘柄コード追加・削除
- 銘柄ごとの閾値スライダー
- 「今すぐスキャン」手動実行
- スキャン結果カード表示

#### サイドバー（全ページ共通）
- ページ選択
- データソース表示
- 場中スキャン コントロール（開始・停止・状態表示）
- 監視銘柄一覧（件数・銘柄コード・銘柄名）

### 銘柄名表示
- ui/utils.py の get_company_name() で取得
- @st.cache_data(ttl=3600) で1時間キャッシュ
- 制限: yfinance の日本語社名精度が低い（例: 三菱商事→三菱株式会社）
- 対応方針: auカブコム API 対応時に正確な名称に置き換える

## 6. Phase 6 計画（未実装）

### 設計思想

誤検知削減と早期検知を独立したテーマとして実装する。

現在（固定閾値モデル）:
  価格変化 >= 閾値  AND  出来高 >= 閾値  →  発報

Phase 6 目標（動的閾値モデル・2段階）:
  Stage 1: 予兆スコアリング（早期検知層）
    → 急騰の予兆を先読みしてスコア化
  Stage 2: スコアに応じて閾値を動的に引き下げ
    → 価格がまだ通常閾値に達していない段階で発報
    + 誤検知削減フィルタで間引く

---

### Phase 6a: バックテスト基盤

目的: 現在の固定閾値モデルの性能を定量評価し、最適閾値を算出する。

実装内容:
- 過去の履歴データに detect_surge() を日付ごとに適用
- 「検知した → 実際にその後N分・N時間上がったか」を集計

算出指標:

| 指標 | 意味 |
|---|---|
| 適合率（Precision） | 発報した中で実際に急騰したものの割合（誤検知の少なさ） |
| 再現率（Recall） | 実際の急騰をどれだけ検知できたか（検知漏れの少なさ） |
| 勝率 | 発報後に買って利益になった割合 |
| 期待値 | 発報1回あたりの平均損益（勝率 × 平均利益 − 負率 × 平均損失） |
| 最適閾値 | 期待値または勝率が最大になる閾値の組み合わせ |

検証テーマ:
- RSI・GC・BBを判定条件に加えると適合率・期待値は改善するか
- 閾値を変えたとき適合率と再現率はどのようにトレードオフするか

---

### Phase 6b: 早期検知層の実装

目的: 価格が固定閾値に達する前の「予兆段階」で急騰を検知する。

実装する指標:

| 指標 | 説明 | 予兆検知への寄与 |
|---|---|---|
| モメンタム（Momentum） | 直近N本の価格変化の絶対値 | 勢いの加速を検知 |
| ROC（Rate of Change） | 価格変化率の変化率（加速度） | 急加速の予兆 |
| 出来高急増速度 | 直近N本の出来高変化率（前足比） | 資金流入の予兆 |

出来高急増速度のアイデア（最重要）:
  現在の「出来高比 >= 3倍（20日平均比）」ではなく
  「直近5本で出来高が急加速している」を条件にすることで
  価格が閾値に達する前に検知できる可能性がある。

---

### Phase 6c: 動的閾値モデルの実装

目的: 予兆スコアに応じて価格閾値を動的に引き下げ、早期発報を実現する。

設計:
  急騰確率スコア = f(モメンタム, ROC, 出来高急増速度)

  スコア低  → 閾値: 通常（前日終値比 3%）
  スコア中  → 閾値: 緩和（前日終値比 2%）
  スコア高  → 閾値: 大幅緩和（前日終値比 1.5%）
                + 誤検知削減フィルタ（RSI > 70 OR BB上抜け）を必須に

  → 価格がまだ3%に達していない段階でアラート発報
  → その後3%→5%と上昇（通常閾値を先取りできた）

誤検知削減フィルタ（Stage 2）:

| フィルタ | 役割 |
|---|---|
| RSI > 70 | 勢いが伴っているか確認（遅行・確認用） |
| BB上抜け | 統計的に異常な動きか確認（遅行・確認用） |
| ゴールデンクロス | 中期トレンド転換の確認（遅行・補助） |

### Phase 6d: auカブコム証券API対応

目的: リアルタイムデータによる根本的な遅延解消。

改善効果:

| 項目 | yfinance（現在） | auカブコム API（将来） |
|---|---|---|
| データ遅延 | 15分 | リアルタイム |
| スキャン間隔 | 5分 | 数秒〜1分 |
| 最大遅延 | 約20分 | 数秒 |
| 社名データ | 不正確（yfinance） | 正確 |
| 足種選択（スキャン） | 15分固定 | 1分・5分・15分を選択可 |

実装対象:
- data/kabucom_fetcher.py（BaseFetcher 実装）
- config.py に DATA_SOURCE = "kabucom" 追加
- ui/utils.py の get_company_name() をauカブコム APIに切り替え
- スキャン足種をUIから選択可能にする

## 7. 既知の制限・技術的負債

| 項目 | 内容 | 解消予定 |
|---|---|---|
| 銘柄名の不正確さ | yfinance の日本語社名が不正確 | Phase 6d |
| データ遅延 | yfinance は15分遅延 | Phase 6d |
| スキャン足種固定 | 15分足のみ（UIで変更不可） | Phase 6d 後 |
| バックテスト未実施 | 閾値のデフォルト値に根拠がない | Phase 6a |
| RSI・GC・BB未使用 | 計算・表示のみ、発報判定に未使用 | Phase 6a 検証後に判断 |
| Streamlit再起動でリセット | 監視銘柄・スキャン状態がメモリのみで永続化されない | 将来的にDB永続化を検討 |
